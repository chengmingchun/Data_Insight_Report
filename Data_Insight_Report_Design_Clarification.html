SYSTEM DESIGN CLARIFICATION

AI 数据分析报告工具

这不是“让 AI 读 CSV 然后随便总结”，而是一条可靠的数据工程链路： 先用确定性代码完成清洗、统计和异常检测，再让 LLM 对结构化结果进行解释。

60%工程能力
40%AI 洞察
3+默认图表
0LLM 参与统计计算
核心原则：代码负责算对，AI 负责说清楚。

1. 系统到底做什么

用户上传订单 CSV 后，系统完成完整的数据分析闭环，而不是只展示几张图。

上传 CSV或内置样例数据
→
清洗校验缺失、重复、非法值
→
统计分析计数、求和、分组、Top-N
→
图表与异常趋势、排行、IQR
结构化结果JSON / DTO
→
LLM 洞察趋势、风险、建议
→
HTML 报告可预览、可下载
→
完整交付代码、测试、样例报告

2. 为什么采用模块化单体

题目要的是工程完整性，不是微服务数量。原型阶段最优解是模块清晰、部署简单、测试方便。

更快交付

一个仓库即可运行

不需要前后端分离、服务注册、消息队列和复杂部署。

更容易验证

核心统计可单测

Loader、Cleaner、Analyzer、Anomaly 都可独立测试。

方便演进

模块边界已经保留

未来可将 FastAPI、数据库、异步任务逐步替换进去。

3. 模块职责

页面只负责交互，不承载业务逻辑；所有分析能力都沉淀在独立模块中。

loader.py

读取 CSV、校验类型、编码、字段完整性。

cleaner.py

处理缺失、重复、非法数值，并生成质量报告。

analyzer.py

计算总览指标、分组统计、Top-N 和透视结果。

anomaly.py

执行规则异常和 IQR 统计异常检测。

charts.py

仅负责把分析结果转换为 Plotly 图表。

insight.py

调用 LLM，总结结构化指标，支持 fallback。

report.py

使用 Jinja2 生成完整 HTML 报告。

orchestrator.py

统一编排整个流程，对 UI 提供单一入口。

4. 技术选型为什么这样定

每个技术都服务于题目目标：快速实现、可靠统计、可视化清晰、AI 能力可控。

技术 承担职责 选择原因 没有选择的方向
Python 3.11+ 主要开发语言 数据分析、图表、LLM 生态成熟，原型效率高 Java 会显著增加 CSV、图表和页面成本
Streamlit 上传、展示、下载 无需单独前端工程，指标卡和图表开箱即用 第一版不做 React + FastAPI
Pandas 清洗、聚合、透视 统计能力成熟，容易校验 不引入 Spark，数据规模不需要
Plotly 交互式图表 可嵌入 Streamlit 和 HTML 报告 不使用复杂前端图表框架
Jinja2 HTML 报告导出 模板清晰、离线可查看、便于评审 PDF 作为加分项，不作为首版依赖
Pydantic 配置和结果 DTO 约束模块之间的数据结构 不对 DataFrame 每行建模，避免性能损耗
OpenAI-Compatible API AI 洞察 便于切换模型服务，降低厂商耦合 不把 SDK 直接写进业务层

5. LLM 的边界

这是整个设计最关键的地方：模型只解释结果，不参与任何核心数值计算。

LLM 可以做

  • 总结整体销售表现
  • 描述趋势和排名
  • 解释异常数据可能代表什么
  • 提出业务建议
  • 把 JSON 指标转成自然语言

LLM 不可以做

  • 读取完整原始 CSV 并自行计算
  • 计算总销售额和平均值
  • 判断 Top-N 排名
  • 计算退款率
  • 生成输入中不存在的新数字
{ "overview": { "valid_orders": 982, "total_sales": 1285300.50, "average_order_value": 1308.86, "refund_rate": 0.043 }, "top_products": [ {"product": "显示器", "sales": 235000} ], "anomalies": { "count": 12, "largest_order": 68000 } }
即使 API Key 缺失或 LLM 调用失败，统计、图表和 HTML 报告也必须正常生成。 AI 摘要改用模板化 fallback，不能让外部模型服务成为系统单点。

6. 脏数据如何处理

设计重点不是“删掉坏数据”，而是明确记录每一类问题以及它是否还能参与部分分析。

有效数据

参与全部分析

订单号、商品、数量、单价、折扣等关键字段均合法。

部分有效

参与部分分析

例如日期非法，但金额字段有效，可参与总销售额，不参与时间趋势。

拒绝数据

不进入核心统计

例如数量为负、单价无效、订单号缺失，但仍保留在拒绝记录中供追踪。

重复订单：保留第一条 缺失地区：填充“未知地区” 缺失折扣：填充 0 非法日期：不参与时间趋势 负数量：标记业务异常 极端金额：保留并标记

7. 设计模式不是为了炫技

每个模式都有明确落点，目的是让模块可替换、可测试、可扩展。

策略模式

清洗规则 / 异常算法 / 洞察生成

不同规则独立实现，可增加新规则，不修改主流程。

适配器模式

统一不同 LLM 服务接口

业务层只调用 generate(payload)，不关心底层厂商 SDK。

工厂模式

根据配置创建 Provider

支持 OpenAI、兼容 API、Mock 和模板 fallback。

门面模式

Orchestrator 对 UI 提供单一入口

页面不直接组合八个模块，减少耦合。

9. Codex 实现顺序

不要一次性生成全部代码。按阶段闭环，每阶段完成后运行测试。

阶段 1：工程骨架

目录、依赖、配置、数据模型、异常类、样例数据、README 初稿。

阶段 2：确定性数据链路

Loader、Cleaner、Analyzer、Anomaly 和对应单元测试。

阶段 3：页面与图表

Streamlit 页面、指标卡、数据质量区、三张核心图。

阶段 4：AI 洞察

Provider 接口、LLM Adapter、严格 Prompt、超时处理和 fallback。

阶段 5：HTML 报告

Jinja2 模板、图表嵌入、报告下载、中文显示验证。

阶段 6：交付完善

README、样例报告、测试通过、演示说明和已知限制。

最终面试表达：这是一个可靠的数据分析系统，AI 只是最后一层解释能力。
